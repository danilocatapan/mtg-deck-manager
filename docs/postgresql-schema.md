# PostgreSQL Schema Mapping

## Raciocinio

Schema H2 encontrado:

- A API usa Hibernate ORM/Panache com H2 em memoria configurado em `backend/src/main/resources/application.properties`: `quarkus.datasource.db-kind=h2`, `jdbc:h2:mem:mtg;DB_CLOSE_DELAY=-1` e `quarkus.hibernate-orm.database.generation=drop-and-create`.
- Nao ha `import.sql`, `schema.sql`, Flyway, Liquibase ou migrations versionadas no backend. O schema real atual e gerado por Hibernate a partir das entidades JPA.
- Entidades persistidas encontradas em `backend/src/main/java/com/mtg/model`:
  - `Deck`, tabela `decks`.
  - `DeckCard`, tabela `deck_cards`.
- `DeckRepository` consulta decks por `ownerId` e por `(id, ownerId)`, entao `owner_id` e uma coluna operacional importante para isolamento por usuario.
- Classes de meta/recomendacao em `service/meta` (`MetaCommander`, `MetaCard`, `MetaDeck`, `MetaDeckCard`, `CommanderMetaProfile`, `MetaSourceStatus`) nao sao entidades JPA e hoje sao carregadas/calculadas em memoria a partir de JSON/adapters offline. Portanto nao viraram tabelas neste script para nao inventar persistencia inexistente.

Tabelas atuais mapeadas do H2/Hibernate:

- `decks`
  - `id`: `Long`, `@Id`, `@GeneratedValue(strategy = IDENTITY)`.
  - `name`: `String`, sem `@Column`, logo Hibernate usa `varchar(255)` por padrao no H2.
  - `commander`: `String`, sem `@Column`, `varchar(255)` por padrao.
  - `owner_id`: `String`, `@Column(name = "owner_id")`, `varchar(255)` por padrao.
  - `color_identity`: `String`, `@Column(name = "color_identity")`, `varchar(255)` por padrao.
  - Relacionamento `@OneToMany(mappedBy = "deck", cascade = ALL, orphanRemoval = true, fetch = EAGER)` para `deck_cards`.
- `deck_cards`
  - `id`: `Long`, `@Id`, `@GeneratedValue(strategy = IDENTITY)`.
  - `name`: `String`, `varchar(255)` por padrao.
  - `quantity`: `int`, `integer not null` por ser primitivo Java.
  - `deck_id`: `@ManyToOne(fetch = LAZY)` + `@JoinColumn(name = "deck_id")`, FK para `decks(id)`. Como `optional=false` nao foi declarado, Hibernate permite nulo no mapeamento atual, embora o service sempre associe as cartas ao deck antes de persistir.

Conversao H2 -> PostgreSQL:

- IDs `Long`/identity foram convertidos para `BIGINT GENERATED BY DEFAULT AS IDENTITY`.
- `String` padrao do Hibernate/H2 foi preservado como `VARCHAR(255)` para manter compatibilidade fiel com a entidade atual. Campos com validacao de service menor que isso (`name`, `commander`, nomes de cartas ate 120) receberam `CHECK` nomeado em vez de reduzir o tipo e arriscar divergencia de schema.
- `int` foi convertido para `INTEGER`.
- Nao ha campos persistidos que exijam `UUID`, `TIMESTAMPTZ`, `JSONB`, `TEXT` ou `NUMERIC` no modelo atual. Esses tipos ficam para uma futura persistencia real de imports brutos, caches de cards, meta datasets ou historico de recomendacoes.
- `color_identity` foi mantido como texto curto, porque o codigo atual persiste/consulta um `String` e normaliza lendo caracteres de cores. Uma normalizacao relacional, array ou JSONB mudaria contrato interno e exigiria mudanca Java/migration propria.

Normalizacao/desnormalizacao:

- O modelo atual e normalizado no limite existente: `decks` guarda o cabecalho do deck e `deck_cards` guarda linhas de carta por deck.
- Nao ha tabela `cards` porque cartas vem da Scryfall sob demanda/cache de aplicacao e `DeckCard` persiste apenas nome e quantidade.
- Nao ha tabelas de meta/recomendacao porque esses dados nao sao persistidos hoje. A documentacao do dominio pede que metadados sejam normalizados quando existirem; no codigo atual eles existem como JSON/records transitorios, nao como estado relacional.
- `color_identity` fica desnormalizado em `decks.color_identity`, refletindo exatamente a entidade `Deck`.

PKs, FKs, constraints e indices:

- PKs nomeadas em `decks(id)` e `deck_cards(id)`.
- FK nomeada `fk_deck_cards_deck_id` com `ON DELETE CASCADE`, equivalente ao ciclo de vida atual (`cascade = ALL` + `orphanRemoval = true`) e segura para delecao direta no banco.
- `deck_cards.deck_id` recebeu indice explicito porque e FK e e usado pelo carregamento da colecao.
- `decks.owner_id` recebeu indice por ser usado em `listByOwner`.
- `(id, owner_id)` recebeu indice para a consulta `findByIdAndOwner` e para manter o padrao de isolamento por usuario eficiente. Apesar de `id` ja ser PK, o indice composto ajuda consultas que validam propriedade no mesmo predicado.
- `deck_cards(deck_id, lower(name))` recebeu unique constraint funcional para impedir duplicidade indevida da mesma carta no mesmo deck. Isso reforca a regra de negocio "nao duplicar carta indevidamente" sem bloquear multiplas copias legitimas, que continuam representadas por `quantity`.
- Checks nomeados reforcam invariantes ja validados no service: deck precisa de nome/comandante/owner, quantidade entre 1 e 99, nomes ate 120, ate 120 entradas por deck e ate 99 cartas no main deck. O limite de 99 cartas conta `quantity` em `deck_cards`, mantendo o comandante fora das linhas do deck como o fluxo atual faz.
- As regras agregadas de quantidade/numero de entradas usam triggers porque PostgreSQL nao permite `CHECK` com agregacao entre linhas. Elas sao defensivas para producao; a API ja valida antes.

Suposicoes necessarias:

- O banco alvo e PostgreSQL 12+.
- As cartas do comandante nao sao armazenadas em `deck_cards`; `decks.commander` e a fonte persistida do comandante.
- Decks existentes em H2 eram efemeros por `drop-and-create`; para producao, a populacao inicial/migracao de dados deve ser tratada separadamente.
- O `owner_id` continuara vindo do token Google/OIDC e sera obrigatorio em producao, embora a entidade JPA atual nao declare `nullable=false`.

## Script Final

```sql
BEGIN;

CREATE TABLE IF NOT EXISTS decks (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY,
    name VARCHAR(255),
    commander VARCHAR(255),
    owner_id VARCHAR(255),
    color_identity VARCHAR(255),
    CONSTRAINT pk_decks PRIMARY KEY (id),
    CONSTRAINT ck_decks_name_required CHECK (name IS NOT NULL AND btrim(name) <> ''),
    CONSTRAINT ck_decks_name_length CHECK (name IS NULL OR char_length(name) <= 120),
    CONSTRAINT ck_decks_commander_required CHECK (commander IS NOT NULL AND btrim(commander) <> ''),
    CONSTRAINT ck_decks_commander_length CHECK (commander IS NULL OR char_length(commander) <= 120),
    CONSTRAINT ck_decks_owner_id_required CHECK (owner_id IS NOT NULL AND btrim(owner_id) <> ''),
    CONSTRAINT ck_decks_color_identity_format CHECK (
        color_identity IS NULL
        OR color_identity = ''
        OR regexp_replace(color_identity, '[^A-Za-z]', '', 'g') ~ '^[WUBRGCwubrgc]*$'
    )
);

CREATE TABLE IF NOT EXISTS deck_cards (
    id BIGINT GENERATED BY DEFAULT AS IDENTITY,
    name VARCHAR(255),
    quantity INTEGER NOT NULL,
    deck_id BIGINT,
    CONSTRAINT pk_deck_cards PRIMARY KEY (id),
    CONSTRAINT fk_deck_cards_deck_id FOREIGN KEY (deck_id) REFERENCES decks (id) ON DELETE CASCADE,
    CONSTRAINT ck_deck_cards_name_required CHECK (name IS NOT NULL AND btrim(name) <> ''),
    CONSTRAINT ck_deck_cards_name_length CHECK (name IS NULL OR char_length(name) <= 120),
    CONSTRAINT ck_deck_cards_quantity_range CHECK (quantity BETWEEN 1 AND 99)
);

CREATE UNIQUE INDEX IF NOT EXISTS ux_deck_cards_deck_id_name_ci
    ON deck_cards (deck_id, lower(name))
    WHERE deck_id IS NOT NULL;

CREATE INDEX IF NOT EXISTS ix_decks_owner_id
    ON decks (owner_id);

CREATE INDEX IF NOT EXISTS ix_decks_id_owner_id
    ON decks (id, owner_id);

CREATE INDEX IF NOT EXISTS ix_decks_commander_ci
    ON decks (lower(commander));

CREATE INDEX IF NOT EXISTS ix_deck_cards_deck_id
    ON deck_cards (deck_id);

CREATE INDEX IF NOT EXISTS ix_deck_cards_name_ci
    ON deck_cards (lower(name));

CREATE OR REPLACE FUNCTION fn_validate_deck_card_limits()
RETURNS trigger
LANGUAGE plpgsql
AS $$
DECLARE
    affected_deck_id BIGINT;
    entry_count INTEGER;
    total_quantity INTEGER;
BEGIN
    IF TG_OP = 'DELETE' THEN
        affected_deck_id := OLD.deck_id;
    ELSE
        affected_deck_id := NEW.deck_id;
    END IF;

    IF affected_deck_id IS NULL THEN
        IF TG_OP = 'DELETE' THEN
            RETURN OLD;
        END IF;
        RETURN NEW;
    END IF;

    SELECT count(*), COALESCE(sum(quantity), 0)
      INTO entry_count, total_quantity
      FROM deck_cards
     WHERE deck_id = affected_deck_id;

    IF entry_count > 120 THEN
        RAISE EXCEPTION 'Deck % has % card entries; maximum is 120.', affected_deck_id, entry_count
            USING ERRCODE = '23514',
                  CONSTRAINT = 'ck_deck_cards_entries_per_deck';
    END IF;

    IF total_quantity > 99 THEN
        RAISE EXCEPTION 'Deck % has % cards; maximum main-deck size is 99.', affected_deck_id, total_quantity
            USING ERRCODE = '23514',
                  CONSTRAINT = 'ck_deck_cards_total_quantity_per_deck';
    END IF;

    IF TG_OP = 'DELETE' THEN
        RETURN OLD;
    END IF;
    RETURN NEW;
END;
$$;

DROP TRIGGER IF EXISTS trg_deck_cards_validate_limits ON deck_cards;

CREATE CONSTRAINT TRIGGER trg_deck_cards_validate_limits
AFTER INSERT OR UPDATE OR DELETE ON deck_cards
DEFERRABLE INITIALLY DEFERRED
FOR EACH ROW
EXECUTE FUNCTION fn_validate_deck_card_limits();

COMMIT;
```
